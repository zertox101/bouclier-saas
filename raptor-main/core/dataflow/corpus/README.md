# Dataflow corpus

Labeled dataflow findings used to measure the validator pipeline.

This corpus is the prerequisite for every PR in the dataflow
sanitizer-bypass work (see `~/design/dataflow-sanitizer-bypass.md`).
Each PR's exit criteria is a corpus delta — precision/recall/F1
against this fixed label set, with a per-FP-category breakdown.

The schema (`Finding`, `GroundTruth`) lives at the package root
(`core/dataflow/finding.py`, `core/dataflow/label.py`); this
directory holds the labeled data the schema describes.

## Layout

```
core/dataflow/corpus/
  README.md                          # this file
  SCHEMA.md                          # human-readable JSON reference
  FIXTURES.md                        # how target source trees are sourced
  SOURCES.md                         # pinned URLs + commit shas (TBD)
  findings/                          # one .json + one .label.json per finding
    iris_e2e_codeql_py-command-injection_001.json
    iris_e2e_codeql_py-command-injection_001.label.json
    ...
```

Run scripts (`run_corpus.py`, `corpus_metrics.py`) ship as task #4
under `core/dataflow/`. Operator-facing dev tooling lives at
`core/dataflow/scripts/corpus-run` and
`core/dataflow/scripts/corpus-metrics` — invoke directly; these are
not framework-internal libexec shims and do not require the
`_RAPTOR_TRUSTED` trust marker.

## Status

PR0 seed: 7 hand-labeled findings drawn from existing in-tree iris
fixtures (`packages/llm_analysis/tests/fixtures/iris_e2e/` and
`iris_e2e_multilang/`). Six TPs across Python / JS / Java / Go / C +
one Semgrep-style FP exhibiting the canonical
`missing_sanitizer_model` class.

PR0 will not ship as final until at least 50 entries with realistic
TP/FP balance — OWASP Benchmark Java (~30 entries, free TP/FP labels)
and hand-labelled Juice Shop / WebGoat (~35 combined) populate via
tasks #5 and #6.

## Pivot gate

`corpus_metrics.py` (task #4) computes the share of false positives
in each `fp_category`. The design rejects investing further in the
sanitizer-bypass feature line if `missing_sanitizer_model` is below
10% of total FPs — the dominant FP category should be the one this
work targets, otherwise we should pivot to whichever class actually
dominates.

## Adding a finding

1. Run a producer (CodeQL, IRIS, Semgrep) against a fixture, or
   construct the `Finding` by hand if reproducing.
2. Convert producer output to a `core.dataflow.Finding` via the
   producer's adapter (`core.dataflow.adapters.codeql`, etc.).
3. Pick a stable `finding_id`. Convention:
   `<fixture>_<producer>_<short-rule-slug>_<NNN>` where
   `short-rule-slug` is a clean derivation from the producer's
   rule_id — typically `<lang>-<vuln-class>` (e.g.
   `js-command-injection`, `cpp-command-line-injection`). Stability
   across reruns matters more than mechanical derivation: corpus
   replay matches labels by exact string.
4. Write `findings/<finding_id>.json` from `Finding.to_json(indent=2)`.
5. Hand-label: write `findings/<finding_id>.label.json` with a
   `core.dataflow.GroundTruth`. For false-positives, choose an
   `fp_category` from the closed set in `SCHEMA.md`. Required fields
   (`finding_id`, `rationale`, `labeler`) must be non-empty;
   `labeled_at` must be ISO `YYYY-MM-DD`.

## Known limitations of the PR0 seed

- **Hand-constructed findings.** The seven seed entries were
  hand-written from fixture inspection, not generated by running a
  real producer. Column positions in particular are best-effort
  approximations until the CodeQL adapter (task #2) lands and
  authoritative SARIF output replaces them.
- **Vacuous pivot gate at 1 FP.** With one FP in the seed, the
  100% `missing_sanitizer_model` share is statistical noise. The
  pivot gate becomes meaningful once tasks #5 and #6 grow the
  corpus to realistic FP volume.

## Schema versions

`Finding.SCHEMA_VERSION` and `GroundTruth.SCHEMA_VERSION` are recorded
in every committed file. A bump in either invalidates the corpus
until the existing files are migrated. The corpus runner refuses to
load a mismatched corpus — it does not silently coerce.
