This document outlines the security testing and hardening plan for the Sentinel AI SaaS platform, following the NCSC, OWASP LLM Top 10, and MITRE ATLAS frameworks.

## 2. Hardening Implementation Plan (Live SaaS)

### A. Immediate "Hard" Controls (In-Progress)
1.  **Tool Allowlist (Deterministic)**: No tool can be run unless its ID and parameters match a strict JSON schema. (See `tools-api/app.py`).
2.  **RBAC enforcement**: Map `app.models.sql.User.role` (analyst vs admin) to tool categories.
    - *Analyst*: `ping_host`, `dns_lookup`, `network_recon`.
    - *Admin*: `sqlmap_scan`, `dir_bruteforce`, `packet_sniffer`.
3.  **Circuit Breaker**: Implement `MAX_AGENT_STEPS` to prevent infinite LLM loops if we move to autonomous agents.

### B. "Soft" / LLM-Layer Controls
1.  **Instruction Hierarchy**: Explicitly delimiting user content in prompts:
    ```json
    {"role": "user", "content": "Analyze the following UNTRUSTED content: [CONTENT]"}
    ```
2.  **Robust Prompt Guard**: Upgrade `backend/app/services/prompt_guard.py` to use a lightweight classifier (e.g. DeBERTa-v3-small) or more complex heuristics.

## 3. Live Red-Teaming Regression Suite

- **Tool**: `pytest` or `k6` scripts simulating the payloads in the matrix.
- **Goal**: 0% Success Rate for P1-P5.
- **Monitoring**: Integration with the `Threat Map` to visualize attacks against the AI itself.
